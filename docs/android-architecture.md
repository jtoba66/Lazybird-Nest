# Nest Android Architecture Blueprint

This blueprint describes the recommended Android implementation for Nest given the current protocol in `docs/protocol-spec-v2.md`.

The goal is not a generic Android app. The goal is a native Android client that is protocol-compatible with the existing web client and materially better at large-file reliability.

## 1. Primary Engineering Goal

Android must be a second implementation of the same protocol, not a reinterpretation of it.

Success means:

- A file encrypted on Android opens on the web
- A file encrypted on the web opens on Android
- Metadata written on Android can be read and updated by the web client
- Shared links generated by either client behave identically

## 2. Recommended Stack

Use:

- Kotlin
- Jetpack Compose
- OkHttp
- Retrofit or Ktor client
- Kotlin Coroutines and Flow
- WorkManager
- Room
- Android Keystore
- Libsodium for Android or an equivalent verified native binding

Avoid for the core app:

- WebView shell as the primary product
- Browser-managed file downloads as the main transfer engine
- Re-implementing crypto primitives by hand

## 3. Core Module Layout

Recommended app modules:

- `core-crypto`
- `core-protocol`
- `core-network`
- `core-database`
- `feature-auth`
- `feature-files`
- `feature-share`
- `feature-settings`
- `transfer-engine`

Responsibilities:

- `core-crypto`: libsodium bridge, Argon2id, AEAD, SecretStream, byte conversions, zeroization helpers
- `core-protocol`: request and response models, metadata schema, share URL parsing, compatibility rules
- `core-network`: authenticated API client, retries, upload and download services
- `core-database`: Room entities for local cache, transfer state, and metadata snapshots
- `transfer-engine`: chunk scheduling, resumable uploads, foreground downloads, reconciliation

## 4. Crypto Implementation Rules

Android crypto must follow the protocol spec exactly.

Non-negotiable rules:

- KDF parameters must come from the server response when present
- `authHash` and `wrappingKey` must be derived exactly from `auth_ || RootKey` and `wrap_ || RootKey`
- All API binary fields must use standard padded base64
- SecretStream chunk headers are 24-byte binary headers, not generic nonces
- Chunked uploads use 128 MiB plaintext segments in the current web implementation
- Internal SecretStream block size is 64 MiB

### 4.1 Key storage strategy

Recommended:

- Store JWT and user preferences in `EncryptedSharedPreferences`
- Keep `MasterKey` in memory only during an unlocked session
- If fast unlock is desired, store a device-local wrapped secret using Android Keystore and gate it behind biometrics or device auth

Do not:

- Persist plaintext file keys on disk
- Persist plaintext master key in a recoverable app database

### 4.2 Memory hygiene

For sensitive byte arrays:

- Prefer `ByteArray`
- Zero buffers after use where practical
- Avoid long-lived copies
- Avoid logging secrets in any form

## 5. Auth And Vault Flow

### 5.1 Login

Android login flow should be:

1. Call `POST /auth/salt`
2. Derive `RootKey`
3. Derive `AuthHash`
4. Call `POST /auth/login`
5. Derive `WrappingKey`
6. Decrypt `MasterKey`
7. Decrypt metadata blob
8. Cache non-sensitive session state locally

### 5.2 Signup

Android signup flow should:

1. Generate salt
2. Derive `RootKey`
3. Derive `AuthHash`
4. Derive `WrappingKey`
5. Generate `MasterKey`
6. Encrypt `MasterKey`
7. Create initial metadata blob `{ "v": 2, "folders": {}, "files": {} }`
8. Generate root folder key and encrypt it under the master key
9. Send the same payload shape the web client sends

## 6. Metadata Strategy

The metadata blob is the logical filesystem map. Android should treat it as critical application state.

Recommended local handling:

- Cache the latest decrypted metadata snapshot in Room for UI reads
- Retain the encrypted server blob metadata version alongside it
- Preserve unknown fields when rewriting metadata
- Prefer merge-safe update helpers over rebuilding the whole structure from scratch

Important:

- `metadata_version` is a server-side metadata revision counter
- Metadata JSON `v` is the protocol/schema version
- Do not confuse them
- The current backend increments `metadata_version` but does not enforce stale-write rejection

## 7. Upload Engine

This is the main area where native Android should outperform the web client.

### 7.1 Current protocol requirements

Current web-compatible behavior:

- Monolithic mode for files smaller than 128 MiB
- Chunked mode for files 128 MiB and larger
- Chunked plaintext segment size: 128 MiB
- Each segment uses an independent SecretStream header
- Each segment is internally encrypted in 64 MiB blocks

### 7.2 Android implementation

Recommended design:

- One transfer row per file in Room
- One child row per upload segment
- `WorkManager` worker per segment or bounded worker pool per file
- Persist segment state, retries, and server file ID
- Resume by calling `GET /files/:id/manifest` and skipping completed indices

Upload worker steps:

1. Resolve target folder key
2. Generate file key
3. Call `/files/upload/init`
4. Slice the file into 128 MiB plaintext segments
5. Encrypt each segment independently
6. Upload each ciphertext segment to `/files/:id/chunk`
7. Call `/files/:id/finish`
8. Update encrypted metadata only after upload success

Native advantage:

- Upload can survive app backgrounding
- Upload can survive process death if state is persisted cleanly

## 8. Download Engine

### 8.1 Protocol realities

There are two download layouts:

- Monolithic encrypted file with embedded SecretStream header
- Chunked file where headers are carried in metadata and ciphertext bodies are streamed separately

### 8.2 Android implementation

Recommended:

- Use a foreground service for large downloads
- Stream decrypted bytes directly to `MediaStore` or a user-selected destination
- Do not buffer whole files in memory
- For chunked downloads, reinitialize SecretStream per chunk using the server-provided `nonce`

Owner download path:

- Fetch `/files/download/:id`
- Decrypt folder key then file key
- Use `/files/raw/:id` for monolithic raw content or chunk routes for deterministic chunk retrieval

Share download path:

- Parse file key from the URL fragment
- Fetch `/files/share/:token`
- Use `/files/share/raw/:token` for monolithic content
- Use `/files/share/:token/chunk/:index` plus `chunks[].nonce` for chunked content

## 9. Share Handling

Android should support the exact same fragment format as the web client:

`/s/<token>#key=<base64>&name=<filename>&mime=<mime>`

Rules:

- `key` is required
- `name` and `mime` are optional hints
- Replace spaces with `+` before base64 decode to mirror current web parsing

## 10. Suggested Local Data Model

Room tables should include at least:

- `session`
- `metadata_snapshot`
- `transfer`
- `transfer_segment`
- `download_job`
- `share_preview_cache`

Useful persisted fields:

- User ID
- JWT
- Metadata version
- Last decrypted metadata snapshot timestamp
- File transfer server IDs
- Segment completion map
- Local destination URI for downloads

## 11. Verification Gates Before Shipping

Android should not be considered protocol-ready until all of the following pass:

1. `docs/test-vectors.json` unit tests pass in Android CI
2. Android signup creates an account readable by the web client
3. Android login can decrypt a vault created on the web
4. Android-uploaded monolithic file downloads and decrypts on web
5. Android-uploaded chunked file downloads and decrypts on web
6. Web-uploaded monolithic file downloads and decrypts on Android
7. Web-uploaded chunked file downloads and decrypts on Android
8. Share links generated on either client open on the other
9. Metadata written on Android preserves unknown fields and stays readable by web

## 12. Recommended Delivery Order

Build order should be:

1. `core-crypto`
2. Protocol and test-vector verification
3. Auth and metadata read path
4. Monolithic upload and download
5. Chunked upload and download
6. Share-link support
7. Offline and background hardening
8. UX polish

## 13. Hard Rules For The Native Team

Do not:

- Change base64 conventions
- Invent a new metadata schema
- Change chunk boundaries without a protocol version change
- Assume the server stores plaintext names or folder paths
- Treat the old draft docs as authoritative

Do:

- Treat `docs/protocol-spec-v2.md` as the contract
- Treat `docs/test-vectors.json` as the unit-test baseline
- Add new vectors before changing any protocol behavior

Current strategy version: `1.2`

## 14. Native UI & UX Specification

The Android UX must feel like a premium, native extension of the Nest brand. It should not look like a "generic Android app" but rather a high-end secure vault.

### 14.1 Visual Theme
- **Theme**: Dark-first (True Black #000000 backgrounds for OLED power savings).
- **Primary Color**: Cyan/Electric Blue (#00E5FF or similar) for interactive elements.
- **Surface Styling**: Use Material 3 `ElevatedCard` with custom `Blur` and subtle `LinearGradient` borders to simulate "Glassmorphism".
- **Typography**: `Inter` or `Outfit` (modern, geometric sans-serif).

### 14.2 Key Screens

#### A. The Vault Unlock (Login)
- **Primary UX**: Centered password field with a "Fluid" reveal animation upon decryption success.
- **Biometric Integration**: Prompt for Fingerprint/Face Unlock immediately if a device-local `WrappingKey` is stored in the Android Keystore.
- **Protocol Feedback**: Subtle text indicating "Deriving Root Key..." or "Unwrapping Master Key..." to emphasize the security math happening under the hood.

#### B. The File Explorer
- **Layout**: Flexible list/grid toggle.
- **Empty State**: A minimalist "No files secured yet" graphic with a prominent "+" FAB (Floating Action Button).
- **Offline States**: Show "Cached" labels for files with local metadata. Dither or greyscale thumbnails for files that aren't locally available.
- **Breadcrumbs**: Use a horizontal scrolling `Row` for navigation depth.

#### C. The Secure Share Receiver
- **Aesthetic**: Mirror the "Private Letter" look from the web (minimalist, centered content).
- **Native Action**: Deep-link handling the `/s/` fragment. If the key is in the fragment, decrypt and show the preview instantly.
- **One-Tap Download**: Use a foreground service so the user can leave the app while the large file is decrypted and saved to `Downloads`.

#### D. The Upload "Command Center"
- **Persistent Progress**: Use a `NotificationChannel` with `PROGRESS_BAR` to show multi-chunk upload status.
- **In-App Queue**: A dedicated screen showing active, pending, and failed transfers with a "Retry All" button.

### 14.3 Micro-Animations
- **Shimmer**: Use shimmers while metadata placeholders are being decrypted.
- **Success States**: Subtle haptic feedback and a "Pulse" animation when a file is successfully secured.

## 15. Handoff Readiness

As of Version `1.2`, this blueprint is considered the normative reference for the Android Engineering team. All development should proceed in accordance with the protocol "freeze" documented in `docs/protocol-spec-v2.md`.
